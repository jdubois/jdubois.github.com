🏁 Les règles

Construisez quelque chose de réel, soigné et prêt pour la démo avant le buzzer — et livrez plus que tous les autres.

⏱️ La contrainte

  • 2 jours, pas une minute de plus
  • Vous devez livrer un vrai produit
  • Impossible de coder toute la nuit et de livrer à l'heure

⚡ Votre avantage

  • Vous avez GitHub Copilot
  • Une flotte d'agents en parallèle, pas une seule fenêtre de chat
  • Ils continuent de coder pendant que vous dormez
Mêmes outils, même horloge pour tous. L'équipe qui les exploite le mieux livre plus large et gagne.

Le soir

2 h de prépa = 8 h de construction sans surveillance.

La nuit

Les agents travaillent en parallèle la nuit. Votre marge double.

Le matin

Triez la récolte de la nuit, mergez les gains, rebriefez pour la journée.

🛠️ Étape 1 : Préparez vos agents · le soir

📜 Mettez-le par écrit · AGENTS.md

  • Définissez les règles du jeu une seule fois
  • Ce que vous construisez · conventions · comment builder & tester
  • Chaque agent le lit d'abord, plus besoin de réexpliquer

✅ Donnez-leur un harnais de tests

  • Une vraie commande build + test qu'ils peuvent lancer
  • Ils vérifient leur propre travail avant de rendre compte
  • Pas de tests = output non fiable

🛡️ Posez des garde-fous

  • Une branche / worktree par tâche
  • Périmètre réduit, objectif clair
  • Rien n'atteint main sans votre relecture

🧩 Découpez le travail

  • Découpez le projet en morceaux indépendants
  • Une tâche chacun : deux features, UI, infra
  • Prêt pour le parallèle = prêt pour la nuit
Astuce : meilleur est le brief, moins vous aurez à surveiller à 2 h du matin.

🌙 Étape 2 : Laissez-les coder toute la nuit · pendant votre sommeil

Lancez plusieurs agents d'un coup. Chacun tourne en boucle sur sa propre branche/worktree : builder, tester, ouvrir une PR, prendre la tâche suivante. Un agent répète cette boucle de nombreuses fois avant l'aube.
🚀 Agent 1
Feature principale : ce qui gagne la démo
boucle ×6 · 6 PRs
🧩 Agent 2
Seconde feature clé, en parallèle
boucle ×5 · 5 PRs
🎨 Agent 3
UI, finitions & landing page
boucle ×7 · 7 PRs
🏗️ Agent 4
Infrastructure : déploiement, CI & data
boucle ×4 · 4 PRs
4 agents en boucle toute la nuit → des dizaines de PRs au matin. Votre débit, ce n'est plus votre clavier. C'est la qualité de vos briefs.

☀️ Étape 3 : Relecture devant un café · le matin

Votre routine du matin

  • Triez la récolte de la nuit : des dizaines de PRs
  • Mergez vite les bonnes
  • Jetez ou réaffectez ce qui n'a pas abouti
  • Récupérez les bons morceaux du reste
  • Rebriefez et relancez pour la journée

Réveillez-vous sur du progrès

Des dizaines de PRs vous attendent : chaque agent a bouclé toute la nuit. Le projet a avancé, pas vous. Puis la boucle recommence : brief → build → relecture.
Si vos boucles tournent bien, vous pouvez les relancer plusieurs fois avant midi.

📋 Le plan en une page

1Écrivez les règles du jeu (AGENTS.md) avant de lancer le moindre agent.
2Pas d'agent sans harnais de tests : donnez-lui un build + test qu'il peut lancer lui-même.
3Une tâche par agent. Périmètre réduit, objectif clair, branche dédiée.
4Lancez-les en parallèle : c'est tout l'intérêt. Ne restez pas sur un seul.
5Laissez-les coder toute la nuit pendant que vous dormez.
6Relisez au matin : mergez les gains, jetez les échecs, réaffectez vite.
7Choisissez le bon modèle pour chaque tâche ; sauvegardez ce qui marche en skills réutilisables.
S'il y a une slide à capturer, c'est celle-ci.

⚠️ Attention à

La dérive du périmètre

« Construis toute l'app » fait divaguer l'agent.
Solution : un objectif précis par tâche.

Pas de tests, pas de confiance

Impossible de lire chaque ligne de dizaines de PRs.
Solution : harnais d'abord, build vert avant merge.

PRs géantes

Une PR de 2 000 lignes est impossible à relire.
Solution : petits lots relisables.

Mauvais modèle

Un modèle faible ne résout pas les tâches dures ; un trop puissant est lent et coûteux.
Solution : le bon modèle pour chaque tâche.
La plupart des échecs ne sont pas la faute de l'agent. C'est un problème de brief.
Voilà le plan.
