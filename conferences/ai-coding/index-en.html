What is BootUI?

A production-grade Spring Boot 4 starter that adds an embedded, local-only developer console to your app.

🧩 Multi-module & deeply integrated

  • 5-module Maven build: core, autoconfigure, starter, ui, sample-app
  • Spring Boot 4 / Java 17, Maven Central publishing, full CI
  • Actuator, Spring Security, Flyway/Liquibase, Hibernate
  • Micrometer/OTLP tracing, GraalVM, OSV scanning, ArchUnit

🖥️ ~40 feature panels

  • An embedded Vue 3 SPA, packaged in the starter
  • Each panel = backend endpoints + frontend view + tests
  • Health, metrics, security advisor, vulnerabilities, tracing…
  • The single biggest source of structural repetition
📦 Open source — github.com/jdubois/boot-ui  •  docs at julien-dubois.com/boot-ui

How long it actually took

With AI — the bottom line

Built through a tagged 1.0.0 release by one developer driving the GitHub Copilot coding agent.

~11 days
calendar time, to 1.0.0
~80–110 h
actual human hands-on effort (≈ 2 intense solo weeks)
~20 PRs/day
~223 merged PRs in ~11 days
⚡ A cadence of ~20 PRs/day with an AI agent co-authoring commits is impossible to achieve by hand. The agent did the overwhelming majority of the typing, scaffolding and test writing — the human dispatched many asynchronous tasks in parallel.

With AI — the evidence

📈 Velocity

  • ~223 merged PRs / ~10.8 days (~264 commits)
  • Commit clock runs ~05:00 → midnight most days
  • Consistent with parallel async agent tasks, not continuous typing

🤖 Authorship pattern

  • "Copilot" is a named commit/PR author (~44 commits)
  • Repo ships .github/copilot-instructions.md + per-panel conventions
  • The workflow was explicitly agent-oriented

🚢 Even on release day

  • VuePress docs site + Overview scanner dashboard
  • Copilot token charts, a proxied-Hikari fix, README/docs refactor
  • A volume of polish that is itself several days of solo work

🧭 Where the human time went

  • Writing prompts, reviewing & merging ~223 PRs
  • Resolving CI failures (Spring Boot 4, Flyway 11, OTLP renames)
  • Review-and-orchestrate, not write-every-line

Why the multiplier is so large

This codebase is unusually well-suited to AI assistance — three reasons:

🔁 Massive repetition

~40 structurally similar panels that an agent clones cheaply — the part that is most expensive to hand-write.

🔌 Broad-but-shallow integration

Many Spring subsystems, each shallow. Without AI the cost is mostly looking things up — exactly what the agent absorbs.

🛡️ Strong guardrails

Multi-module CI, CodeQL, e2e tests, explicit copilot-instructions — they let the human safely accept high agent throughput.

🚀 Net effect: an estimated 6.5–8.5 month solo effort compressed into ~11 calendar days and ~2 weeks of human attention — ~12–17× fewer human-hours, ~17–23× faster calendar.

Workflow & token usage

💻 GitHub Copilot App (MacBook Pro)

  • Run many agents in parallel
  • Test changes live before validating

📱 GitHub mobile app

  • Agents in Dockerized containers — prototype on the go
  • Inflates the commit count (quick ideas typed from a phone) — real hours/day are lower

🧠 Models used

  • GPT-5.5 Extra High Reasoning — most of the work, ~1.5M tokens (90% cached)
  • Claude Opus 4.8 & Gemini 3.1 Pro High — the trickiest parts, 3 models complementing each other
  • A smaller model / "Auto" mode for simpler tasks
💰 Total: 120M input + 15M output + 2,500M cached = ~2,700M tokens, for a total cost of ~$2,000.

Conclusion

📊 The numbers

  • ~83k lines, 5 modules, ~40 panels, ~116 test suites, 1.0.0 + docs site
  • With AI: ~11 days, ~80–110 h of human effort
  • Without AI: ~6.5–8.5 months, ~1,100–1,450 h

💡 The lasting lesson

  • AI changed the role from author to architect-and-reviewer
  • Biggest leverage: large-surface, pattern-heavy, well-tested code — not hard algorithms
  • Human judgment stays the irreplaceable bottleneck
